The Complete Guide to Finite State Machine (FSM) Design & Verilog Code

In the world of digital electronics, sequential logic forms the backbone of complex systems, enabling circuits to remember past events and make decisions based on a sequence of inputs. Among the most fundamental and powerful tools for designing such systems are Finite State Machines (FSMs). Whether you're an engineering student grappling with your first digital design course or a hobbyist looking to build intelligent circuits, understanding FSMs is crucial. This guide will take you from the basics of FSMs to practical Verilog implementation, covering everything you need to become a proficient FSM designer.

What is a Finite State Machine (FSM)?

A Finite State Machine (FSM) is a mathematical model of computation used to design sequential logic circuits. At any given time, an FSM can be in one of a finite number of "states." It can change from one state to another in response to external "inputs," and these changes are called "transitions." Depending on its current state and/or inputs, an FSM can also produce "outputs."

Think of an FSM as a system with memory that behaves differently based on its history (current state) and current conditions (inputs). This makes FSMs ideal for controlling sequences of operations, protocols, and decision-making logic in digital circuits.

Key Components of an FSM:

  • States: Discrete operating conditions or modes of the system. Each state represents a unique configuration or memory of past events.
  • Inputs: External signals that influence the state transitions and/or outputs.
  • Outputs: Signals generated by the FSM, often based on the current state and sometimes on the current inputs.
  • Transitions: Rules that define how the FSM moves from one state to another based on specific input conditions.
  • Initial State: The state the FSM starts in when reset or powered on.

Representing FSMs: State Diagrams and State Tables

Before writing any code, it's essential to clearly define the FSM's behavior. Two primary tools help us visualize and organize this information:

State Diagram

A state diagram is a graphical representation of an FSM. It uses:

  • Circles (or nodes): To represent states.
  • Arrows (or directed edges): To represent transitions between states. Each arrow is labeled with the input condition that triggers the transition.
  • Labels within circles: To indicate the outputs generated while in that state (for Moore machines).
  • Labels on arrows: To indicate outputs generated during the transition (for Mealy machines).

State Table

A state table is a tabular representation that lists all possible states, inputs, next states, and outputs. It provides a more structured and exhaustive view of the FSM's behavior, which is particularly useful for deriving the logic equations during implementation.

A typical state table includes columns for:

  • Current State
  • Input(s)
  • Next State
  • Output(s)

Types of Finite State Machines: Mealy vs. Moore Machines

FSMs are broadly categorized into two main types based on how they generate outputs: Mealy machines and Moore machines. Understanding their differences is critical for choosing the right model for your design.

Moore Machine

In a Moore machine, the outputs depend only on the current state. This means that as long as the FSM is in a particular state, its outputs remain constant, regardless of the inputs.

  • Characteristics:
    • Outputs are associated with states.
    • Outputs change synchronously with state changes (typically on a clock edge).
    • Generally considered "safer" because outputs are stable for an entire clock cycle, preventing glitches due to input changes.
  • Advantages: Simpler output logic, outputs are stable, easier to debug.
  • Disadvantages: May require more states to achieve the same functionality as a Mealy machine, outputs can be delayed by one clock cycle relative to the input that caused the state change.

Mealy Machine

In a Mealy machine, the outputs depend on both the current state and the current inputs. This allows for more immediate responses to inputs, but also introduces the possibility of output glitches if inputs change asynchronously.

  • Characteristics:
    • Outputs are associated with transitions (arrows) between states.
    • Outputs can change immediately when an input changes, even if the state hasn't changed yet.
  • Advantages: Can often achieve the desired functionality with fewer states than a Moore machine, outputs respond faster to inputs.
  • Disadvantages: Output logic can be more complex, outputs might produce glitches if inputs change asynchronously or are not synchronized with the clock, harder to debug.

Mealy vs. Moore Comparison

Here's a quick comparison:

Feature         Mealy Machine                   Moore Machine
----------------------------------------------------------------------------------
Output Logic    Current State AND Current Input   Current State ONLY
Output Timing   Asynchronous (can change mid-cycle) Synchronous (changes with state)
Complexity      Potentially fewer states, more   Potentially more states, simpler
                complex output logic             output logic
Glitches        More prone to output glitches    Less prone to output glitches
Responsiveness  Faster response to inputs        Slower response (1 clock delay)

The FSM Design Process: A Step-by-Step Guide for the FSM Designer

Designing an FSM, whether for a simple counter or a complex communication protocol, follows a systematic approach. Here's the typical workflow:

Step 1: Understand the Problem Statement and Requirements

Clearly define what the FSM needs to do. Identify all possible inputs, desired outputs, and the sequence of operations. What events trigger what actions? What are the initial conditions?

Step 2: Define States

Based on the problem statement, identify all unique operating modes or conditions the system can be in. Each state should represent a distinct memory of past events or a specific phase of operation.

Step 3: Draw the State Diagram

Visually map out the states and transitions. This is often the most intuitive step.

  • Draw a circle for each state.
  • Draw arrows between states to represent transitions.
  • Label each arrow with the input condition that causes the transition.
  • For Moore machines, label the outputs inside the state circles.
  • For Mealy machines, label the outputs on the transition arrows (Input/Output).
  • Indicate the initial (reset) state.

Step 4: Create the State Table

Convert your state diagram into a tabular format. This ensures you've considered all possible combinations of current states and inputs, and explicitly defined the next state and outputs for each.

Step 5: State Assignment (Encoding)

Assign unique binary codes to each state. The choice of encoding can impact the complexity of the combinational logic for next state and outputs, and thus the area, speed, and power consumption of the final circuit. Common encoding schemes include:

  • Binary (Sequential): Assigns states in a simple binary sequence (e.g., S0=00, S1=01, S2=10). Minimizes flip-flop count.
  • One-Hot: Uses one flip-flop per state, where only one flip-flop is '1' at any time (e.g., S0=0001, S1=0010, S2=0100). Requires more flip-flops but often results in simpler next-state logic and faster operation.
  • Gray Code: Ensures only one bit changes between adjacent states, which can reduce transient hazards.

Step 6: Derive Next-State and Output Logic

Using the state table and state assignments, derive the Boolean equations for the next state of each flip-flop and for each output. Karnaugh Maps (K-maps) or other logic minimization techniques are often used here.

Step 7: Implementation (Verilog HDL)

Translate the derived logic into hardware description language (HDL), such as Verilog. This involves defining states, implementing the combinational next-state logic, the sequential state update logic, and the output logic.

Practical Example: A Simple Traffic Light Controller (Moore Machine)

Let's design a simple traffic light controller for a two-way intersection (North/South and East/West). The lights cycle automatically based on a timer. For simplicity, we'll assume a single input `timer_tick` that goes high when a predefined time interval has passed.

Problem Description

The traffic light should cycle through the following sequence:

  1. North/South Green, East/West Red (for a long duration)
  2. North/South Yellow, East/West Red (for a short duration)
  3. North/South Red, East/West Green (for a long duration)
  4. North/South Red, East/West Yellow (for a short duration)
  5. Repeat

Input: `clk`, `reset`, `timer_tick` (pulse high when time for transition).
Outputs: `ns_green_o`, `ns_yellow_o`, `ew_green_o`, `ew_yellow_o` (active high).

Step 1 & 2: State Definition

We need four states:

  • NS_GREEN: North/South traffic has green light.
  • NS_YELLOW: North/South traffic has yellow light.
  • EW_GREEN: East/West traffic has green light.
  • EW_YELLOW: East/West traffic has yellow light.

Step 3: State Diagram

Since this is a Moore machine, outputs are associated with states.


        +-----------+    timer_tick    +-----------+
        |           |  ------------>  |           |
        | NS_GREEN  |                 | NS_YELLOW |
        | NS_G=1    |                 | NS_Y=1    |
        | EW_R=1    |                 | EW_R=1    |
        +-----------+                 +-----------+
              ^                             |
              | timer_tick                  | timer_tick
              |                             |
        +-----------+                 +-----------+
        |           |  <------------  |           |
        | EW_YELLOW |    timer_tick   | EW_GREEN  |
        | EW_Y=1    |                 | EW_G=1    |
        | NS_R=1    |                 | NS_R=1    |
        +-----------+                 +-----------+

(Note: Red lights are implied when the other direction is not green/yellow).

Step 4: State Table

Let's assign 2-bit binary codes for the states (e.g., NS_GREEN=00, NS_YELLOW=01, EW_GREEN=10, EW_YELLOW=11).

NS_GREEN (00) 0 NS_GREEN (00) 1 0 0 0
NS_GREEN (00) 1 NS_YELLOW (01) 1 0 0 0
NS_YELLOW (01) 0 NS_YELLOW (01) 0 1 0 0
NS_YELLOW (01) 1 EW_GREEN (10) 0 1 0 0
EW_GREEN (10) 0 EW_GREEN (10) 0 0 1 0
EW_GREEN (10) 1 EW_YELLOW (11) 0 0 1 0
EW_YELLOW (11) 0 EW_YELLOW (11) 0 0 0 1
EW_YELLOW (11) 1 NS_GREEN (00) 0 0 0 1

Step 5 & 6: State Assignment and Logic Derivation

This step would typically involve K-maps for deriving logic from the state table. For Verilog, we can directly implement the logic using `case` statements, which often simplifies this manual derivation.

Step 7: Verilog Implementation (Finite State Machine Verilog)

Here's how to implement this Moore machine in Verilog using the common "three-segment" FSM style:

  1. State Register: A sequential block to update the current state at each clock edge.
  2. Next-State Logic: A combinational block to determine the next state based on the current state and inputs.
  3. Output Logic: A combinational block to determine outputs based on the current state (and inputs for Mealy).

module traffic_light_fsm (
    input wire        clk,
    input wire        reset_n,    // Asynchronous active-low reset
    input wire        timer_tick, // Pulse high when time to transition

    output reg        ns_green_o,
    output reg        ns_yellow_o,
    output reg        ew_green_o,
    output reg        ew_yellow_o
);

    // 1. State Parameter Definition (for readability and maintainability)
    localparam [1:0]
        NS_GREEN  = 2'b00,
        NS_YELLOW = 2'b01,
        EW_GREEN  = 2'b10,
        EW_YELLOW = 2'b11;

    // State Variables
    reg [1:0] current_state;
    reg [1:0] next_state;

    // 2. State Register - Sequential Logic (State Update)
    always @(posedge clk or negedge reset_n) begin
        if (!reset_n) begin
            current_state <= NS_GREEN; // Reset to initial state
        end else begin
            current_state <= next_state;
        end
    end

    // 3. Next-State Logic - Combinational Logic
    always @(*) begin
        next_state = current_state; // Default: stay in current state

        case (current_state)
            NS_GREEN: begin
                if (timer_tick)
                    next_state = NS_YELLOW;
            end
            NS_YELLOW: begin
                if (timer_tick)
                    next_state = EW_GREEN;
            end
            EW_GREEN: begin
                if (timer_tick)
                    next_state = EW_YELLOW;
            end
            EW_YELLOW: begin
                if (timer_tick)
                    next_state = NS_GREEN;
            end
            default: begin
                // Handle illegal states - go to a known safe state
                next_state = NS_GREEN;
            end
        endcase
    end

    // 4. Output Logic - Combinational Logic (Moore Machine)
    // Outputs depend ONLY on the current state
    always @(*) begin
        ns_green_o  = 1'b0;
        ns_yellow_o = 1'b0;
        ew_green_o  = 1'b0;
        ew_yellow_o = 1'b0;

        case (current_state)
            NS_GREEN: begin
                ns_green_o = 1'b1;
            end
            NS_YELLOW: begin
                ns_yellow_o = 1'b1;
            end
            EW_GREEN: begin
                ew_green_o = 1'b1;
            end
            EW_YELLOW: begin
                ew_yellow_o = 1'b1;
            end
            default: begin
                // Default outputs for illegal states (e.g., all red)
                ns_green_o  = 1'b0;
                ns_yellow_o = 1'b0;
                ew_green_o  = 1'b0;
                ew_yellow_o = 1'b0;
            end
        endcase
    end

endmodule

Verilog Coding Best Practices for FSMs

To write robust, synthesizable, and readable finite state machine Verilog code, follow these best practices:

  • Use parameter for State Encoding: Define states using localparam or parameter for clarity and easy modification. Avoid hardcoding binary values.
  • Three-Segment FSM Style: Separate your FSM into three distinct `always` blocks:
    1. One for synchronous state updates (always @(posedge clk or negedge reset_n)).
    2. One for combinational next-state logic (always @(*)).
    3. One for combinational output logic (always @(*)).
    This style is highly recommended for clarity, synthesizability, and avoiding common pitfalls like latches.
  • Synchronous State Updates: Always update the `current_state` register on a clock edge.
  • Reset Logic: Include a reset mechanism (asynchronous or synchronous) to ensure the FSM starts in a known state. Active-low resets are common.
  • Full Case Coverage: Use `default` cases in `case` statements for both next-state and output logic. This prevents unintended latches (in combinational blocks) and ensures defined behavior for unexpected state values (e.g., during power-up or due to radiation effects).
  • Non-Blocking vs. Blocking Assignments: Use non-blocking assignments (`<=`) for sequential logic (state updates) and blocking assignments (`=`) for combinational logic (next-state and output logic within `always @(*)` blocks).
  • Avoid Latches: Ensure all outputs of a combinational `always @(*)` block are assigned a value under all possible conditions to avoid inferring unwanted latches. The `default` case helps here.

Advanced FSM Concepts (Brief Overview)

  • Hierarchical FSMs (HFSMs): For very complex systems, FSMs can be nested within other FSMs, allowing for modular design and easier management of complexity.
  • FSM with Datapath: Many real-world systems combine an FSM (control unit) with a datapath (data processing unit). The FSM controls the operations of the datapath.
  • Handling Metastability: When asynchronous inputs interact with synchronous FSMs, metastability can occur. Techniques like using synchronizer flip-flops are crucial to mitigate this risk.

Conclusion

Finite State Machines are an indispensable concept in digital electronics state machine design. By systematically defining states, transitions, inputs, and outputs, you can model and implement complex sequential behaviors with clarity and precision. Whether you choose a Mealy or Moore architecture, following a structured design process and adhering to Verilog best practices will lead to robust, efficient, and maintainable designs.

As an aspiring FSM designer, remember that practice is key. Start with simple designs, draw your state diagrams, build your state tables, and then translate them into Verilog. Experiment with different FSM types and state encoding schemes. The more you practice, the more intuitive FSM design will become, opening up a world of possibilities in your digital circuit endeavors.
